# Audit Workflow Docs

Goal: make a Flow's **autogenerated** reference page a superset of the old
hand-written doc, so the hand-written one can be deleted. All fixes go in
**source** (TSDoc, barrel exports, `GUIDE.md`) — never hand-edit files under
`docs/reference/` (they are regenerated and overwritten).

## 1. Locate the pieces

- Source: `src/components/<Domain>/<Flow>/` and its `*StateMachine.ts`.
- Generated page(s): `docs/reference/<domain>/<flow>-flow.md` + `blocks.md`.
- Old hand-written doc (if any): `docs/workflows-overview/<name>.md` — the thing to delete.

## 2. Diff old vs autogen

Props tables, event tables, examples, and subcomponent listings are almost
always **already richer** in the autogen — do not migrate them. Hunt only for
**guide-only** content: the step sequence / journey, branch conditions, data
dependencies, business-rule callouts. Ignore stale content in the old doc
(wrong prop names, outdated SCREAMING_SNAKE event names) — the autogen is
authoritative.

## 3. Checklist

- [ ] **Props render** — the page shows a real `## <Flow>Props` table, not a
      bare `## Parameters` row. If bare: the `*Props` (and `*DefaultValues`) type
      isn't re-exported from the component's barrel, so `excludeNotDocumented`
      dropped it. Add `type <Flow>Props` to the `exports/*.ts` barrel and the
      component's `index.ts`. TypeDoc even warns "export it."
- [ ] **`@components` tag** — the Flow's TSDoc lists its sub-components as
      `@components` with `{@link}` items. Renders as a **Sub-components table** on
      flow pages, a **list** on block components. It's its own tag group (see
      `eslint-rules/tsdoc-sort-tags.ts`); place it after `@remarks`, blank-line
      separated. Trim any redundant `{@link}` enumeration out of `@remarks`.
- [ ] **Step-flow diagram** — a `GUIDE.md` next to the component (`overview` or
      `appendix` slot) holds a Mermaid `flowchart` drawn from the **actual state
      machine**. See "Diagrams" below.
- [ ] **Guide-only prose** — anything narrative the autogen can't derive lives
      in `GUIDE.md` (flow level) or `@remarks` (component level).

## 4. Diagrams (Mermaid)

Mermaid is enabled (`docs-site/docusaurus.config.ts`). Author in `GUIDE.md`.

- Read the `*StateMachine.ts` for real transitions; pull event strings from
  `src/shared/constants.ts` (`componentEvents`) — they are not guessable
  (e.g. `employee/compensations/done`, return is `company/employees`).
- Keep diagrams **linear**. If a machine branches (e.g. self-onboarding skips
  steps), render **one diagram per path** rather than one tangled diagram.
- Use a hexagon `id{{"flag?"}}` for a guard-driven routing decision, with
  `true`/`false` branches — distinct from event-labeled edges.
- Show the terminal "route away" event as an edge to a tiny circle `done(( ))`.
- Don't force layout (`direction`); let Mermaid decide. Subgraph `direction` is
  ignored when an outside node links in.

## 5. Verify

```bash
cd docs-site && npm run typedoc   # regenerates docs/reference/
```

Regeneration must report **0 errors**. Spot-check the page renders the props table,
Sub-components, and the diagram(s). Run the docs dev server (`npm run start` in
`docs-site`) to confirm Mermaid renders — but check whether the user already
has one running first.
